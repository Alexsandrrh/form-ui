# Contributing Guide

Документ описывает процесс изменения схем и обязательное сопровождение документации.

## Базовый workflow

1. Измените нужные схемы в `src/schemas/`.
2. Если меняется логика `validation`, обновите соответствующий `form-ui-*-field-validation.ts`.
3. Синхронизируйте публичные экспорты в `src/schemas/index.ts`.
4. Обновите документацию (минимум: `schema-reference`, `examples`, ссылки в `README.md`).
5. Обновите `use-cases/`, если изменился контракт или рекомендуемый паттерн использования.
6. Запустите проверки:
   - `bun test`
   - `bunx tsc --noEmit`

## Требования к схемам

При добавлении/изменении схемы придерживайтесь правил репозитория:
- имя файла: `form-ui-*.ts`;
- экспорт схемы: `...Schema` с `meta({ id, description })`;
- экспорт типа: `...` через `z.infer<typeof ...Schema>`;
- русскоязычные `description` в `meta`;
- для валидаций полей использовать отдельные модули `form-ui-*-field-validation.ts`.

## Источник правды по API

При ревью документации источником правды по публичному API считается:
- `src/schemas/index.ts` — полный список экспортируемых сущностей;
- конкретные `src/schemas/form-ui-*.ts` — точные ограничения, дефолты и shape.

Если референс расходится с кодом, корректным считается код, документацию нужно обновить в этом же PR.

## Обязательный чеклист при изменении схем

- [ ] Обновлён код схем в `src/schemas/`.
- [ ] При изменении правил поля обновлён соответствующий `form-ui-*-field-validation.ts`.
- [ ] Синхронизирован barrel `src/schemas/index.ts`.
- [ ] Обновлён [schema-reference.md](./schema-reference.md): поля, типы `validation`, дефолты, ограничения.
- [ ] Обновлён [examples.md](./examples.md): валидные и невалидные примеры под новый контракт.
- [ ] В тестах есть проверка strict-поведения (неподдерживаемые ключи отклоняются).
- [ ] Проверены ссылки в `README.md` на документацию.
- [ ] При необходимости обновлён `use-cases/reports.ts`.
- [ ] Выполнен `bun test`.
- [ ] Выполнен `bunx tsc --noEmit`.

## Коммиты и PR

Используйте Conventional Commits:
- формат: `type(scope): описание`
- типы: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`
- описание после `:` — на русском, в действии

Для PR добавляйте:
- краткое описание изменений;
- результаты проверок (`bun test`, `bunx tsc --noEmit`);
- при изменении контракта схем — какие разделы документации обновлены.
